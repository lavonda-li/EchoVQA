     Plan: EchoVQA Video Linking Pipeline

     Context

     Each entry in mimic_all_tiers.json is a clinical Q&A session grounded 
     in an echocardiography report. To turn it into a video-based benchmark,
      each question must be tagged with the specific DICOM video clips that
     could be used to answer it — just as a cardiologist would pull up only
     the relevant clips during review.

     Data hierarchy:
     - One study_id = one patient visit
     - Each study has 20–30 DICOM video clips
     - Each clip shows the heart from a specific angle (view: A4C, PLAX, 
     PSAX, etc.)
     - A question about "Mitral Regurgitation" needs only clips showing the 
     mitral valve

     Why keyword matching partially fails:
     - ~59% of entries have full template-built reports → exact substring 
     match works
     - ~41% have condensed summaries (avg 461 chars vs 1,236 for full 
     reports) → no dictionary phrases appear verbatim
     - The 41% are valid data with real Q&A pairs and available videos — 
     they just can't be view-filtered
     - Decision: For non-matching entries, fall back to all qualifying study
      videos (score > 0.7)

● Here's the full workflow from start to finish:

  ---
  Goal: Each Q&A entry in mimic_all_tiers.json needed a videos field pointing to the specific echocardiogram clips a model would need to watch to answer the question.

  ---
  Step 1 — Understand the data

  Explored all four files to understand their structure and how they connect:
  - mimic_all_tiers.json — 1,997 clinical Q&A entries, each tied to a patient study via study_id
  - echo_data_dictionary.xlsx — 2,453 standardized echo report phrases, each with an impression_id
  - ViewStatementMapping-newVC-6-26-25.xlsx — maps 217 impression_ids to the echo views (A4C, PLAX, etc.) needed to assess each finding
  - preprocessed_view_classification_1113.csv — 246K rows of video files with their classified view and confidence score

  Step 2 — Design the linking chain

  The 4-hop chain connecting text to video:
  report text → impression_id → required echo view → video file path

  Step 3 — Discover the keyword matching problem

  Initial plan was exact substring matching: strip [[MEAS]]/[[MOD]] placeholders from dictionary phrases, then check if the phrase appears verbatim in the report. Testing showed this only worked for 59% of entries — the other 41% use condensed summaries that don't contain the exact template phrases.

  Step 4 — Switch to semantic matching

  Replaced substring matching with OpenAI embeddings (text-embedding-3-small):
  - Embed all 216 relevant dictionary phrases once, cache to disk
  - Embed all 1,997 report texts, cache to disk
  - At runtime: compute cosine similarity between each report and all phrases; collect impression_ids above a 0.45 threshold

  This raised coverage from 59% → 99.8%.

  Step 5 — Map views to videos

  For each entry's matched impression_ids:
  1. Look up which echo views are required in ViewStatementMapping
  2. Filter the CSV to rows matching that study_id + required view + confidence > 0.7
  3. Write the sorted, deduplicated file paths as the videos field

  Step 6 — Handle the view name mismatch

  The XLSX and CSV used different naming conventions for the same views (e.g. PSAX- apex vs PSAX-Apex, Color-Aortic-Regurgitation vs color-Aortic-Regurgitantion — including a typo in the CSV). Built an explicit normalization map to bridge them.

  Step 7 — Final output

  mimic_all_tiers.json updated with:
  - videos: list of qualifying video file paths
  - video_match_method: "semantic" or "no_match"

  Results: 1,994/1,997 entries matched, 78,141 total video paths assigned, mean ~39 videos per entry.

  Step 8 — Committed and pushed

  Pushed link_videos.py, updated mimic_all_tiers.json, CLAUDE.md (documentation), embedding cache index files, and updated .gitignore to exclude the 74MB CSV and large .npy files.
